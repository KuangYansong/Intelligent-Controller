# 智能固态断路器源码入门讲解

这份文档不是简单复述每一行代码，而是按“文件职责 + 关键函数 + 执行顺序”来解释，方便代码初学者先建立整体理解，再回头看具体实现。

## 1. 这个项目在做什么

这个项目是“智能固态断路器”的第一阶段固件框架，运行在 TI 的 `TMS320F280049C` 控制芯片上。

它主要完成 4 件事：

1. 采样电压、电流、温度。
2. 判断是否过压、过温、过载、短路。
3. 通过 PWM / 比较器控制断路器是否跳闸。
4. 通过 CAN 总线和上位机通信。

同时，项目里还有一套上位机 C# 代码，用来解析协议、测试协议和做桌面界面。

## 2. 先从哪里开始看

如果你是初学者，建议按下面顺序阅读：

1. `main.c`
2. `app/system.h`
3. `app/system.c`
4. `bsp/board.c`
5. `protection/protection.c`
6. `protocol/protocol_service.c`
7. `storage/param_store.c`
8. `driver/*.c`
9. `host/` 目录下的 C# 代码

原因很简单：

- `main.c` 负责入口。
- `app/system.c` 负责主循环调度。
- 其他模块都是被它调用的“功能件”。

## 3. 主流程怎么跑

### 3.1 程序入口 `main.c`

`main()` 是程序起点。

它只做两件事：

1. 创建一个 `SystemContext system` 变量。
2. 调用 `System_Init(&system)` 初始化系统。
3. 然后在死循环里一直调用 `System_RunOnce(&system)`。

你可以把它理解成：

- `System_Init()`：开机准备。
- `System_RunOnce()`：每转一圈，执行一次系统任务。

### 3.2 系统总控 `app/system.c`

这是整个固件最核心的文件。

`SystemContext` 结构体里保存了系统运行需要的大部分状态：

- 参数 `params`
- 原始 ADC 采样 `raw`
- 换算后的测量值 `measurements`
- 保护服务 `protection`
- 协议服务 `protocol`
- 各类定时任务的时间戳

`System_Init()` 的工作顺序大致是：

1. 初始化时间基准 `Timebase_Init()`
2. 初始化板级外设 `Board_Init()`
3. 从 FRAM 读取参数 `ParamStore_Load()`
4. 初始化故障日志 `FaultLog_Init()`
5. 初始化短路比较器 `CmpssDriver_Init()`
6. 初始化 CAN 驱动 `CanDriver_Init()`
7. 读取一次 ADC 采样
8. 初始化保护模块 `Protection_Init()`
9. 初始化协议模块 `ProtocolService_Init()`
10. 执行上电自检 `SelfTest_RunPowerOn()`
11. 记录定时任务的起始时间

`System_RunOnce()` 每次循环会做这些事：

1. 读取最新 ADC 数据
2. 换算成电压/电流/温度
3. 更新 RMS 电流
4. 轮询 CAN 收包
5. 每 10ms 做一次保护判断和周期自检
6. 定时发送实时数据报文
7. 定时发送心跳报文
8. 进入空闲态

## 4. 各目录是做什么的

### 4.1 `app`

这是“应用层”。

- `app/system.h`
  说明系统上下文 `SystemContext` 长什么样。
- `app/system.c`
  负责把所有模块串起来，像项目里的“总调度员”。

### 4.2 `bsp`

这是板级支持包，可以理解为“离硬件最近的一层总开关”。

- `bsp/board.c`
  做底层初始化，包括：
  - 芯片启动初始化
  - GPIO 初始化
  - 中断模块初始化
  - ADC / PWM / FRAM 初始化

关键函数：

- `Board_Init()`：初始化整块板子的基础外设
- `Board_Idle()`：没有任务时进入空闲
- `Board_SoftwareReset()`：软件复位芯片

### 4.3 `driver`

这是驱动层，对单个外设做封装。

- `adc_driver.c`
  读取 ADC 原始值，并换算成工程量。
- `epwm_driver.c`
  控制 PWM 输出，也负责跳闸输出。
- `cmpss_driver.c`
  设置短路比较器阈值，处理快速硬件保护。
- `can_driver.c`
  CAN 初始化、发送、轮询接收。
- `spi_driver.c`
  给 FRAM 存储器提供 SPI 读写。

你可以把这一层理解成：

- 上层只关心“我要读取电流”。
- 驱动层负责处理“具体寄存器怎么配”。

### 4.4 `protection`

这是保护逻辑层，是断路器项目最重要的业务逻辑之一。

- `protection.c`
  总保护状态机。
- `rms.c`
  计算电流有效值 RMS。
- `i2t.c`
  计算 I²T 积分，用来判断过载累积是否超标。

关键逻辑：

- `Protection_Evaluate10ms()`
  每 10ms 判断一次：
  - 采样值是否合法
  - 是否过压
  - 是否过温
  - 是否过载

- `Protection_OnShortTrip()`
  用于短路快速跳闸。

- `Protection_UpdateRecovery()`
  决定故障后能不能恢复、什么时候恢复。

这部分可以理解成断路器的“大脑”：

- 正常就放行。
- 出故障就跳闸。
- 条件满足时再考虑恢复。

### 4.5 `protocol`

这是通信协议层，负责和上位机说话。

- `can_protocol.c`
  负责“报文格式”：
  - 怎么拼一帧实时数据
  - 怎么拼一帧故障数据
  - 怎么解析控制命令
- `protocol_service.c`
  负责“业务动作”：
  - 收到时间同步命令怎么办
  - 收到参数读写命令怎么办
  - 收到控制命令怎么办

关键函数：

- `ProtocolService_OnRx()`
  收到 CAN 报文后的统一入口。
- `ProtocolService_SendRealtime()`
  发送实时测量值。
- `ProtocolService_SendFault()`
  发送故障记录。
- `ProtocolService_SendHeartbeat()`
  发送心跳。

### 4.6 `storage`

这是存储层，负责参数和故障记录。

- `fram.c`
  FRAM 的更高一层封装。
- `param_store.c`
  保存和读取系统参数。
- `fault_log.c`
  保存和读取故障日志。

`param_store.c` 很值得初学者看，因为它展示了嵌入式里很常见的“参数持久化”套路：

1. 上电从非易失存储器读参数。
2. 检查 `magic`、`version`、`crc16` 是否正确。
3. 如果损坏，就恢复默认值。
4. 修改参数后重新写回 FRAM。

### 4.7 `selftest`

这是自检层。

- `SelfTest_RunPowerOn()`
  上电时跑一次自检。
- `SelfTest_RunPeriodic()`
  运行过程中周期性自检。
- `SelfTest_Passed()`
  判断自检是否通过。

### 4.8 `common`

这是公共工具层。

- `sscb_types.h`
  定义全项目共用的数据结构和枚举。
- `sscb_config.h`
  定义配置常量、地址、默认阈值。
- `crc16.c`
  算 CRC 校验。
- `timebase.c`
  管理毫秒时间和 Unix 时间。
- `utils.h`
  小工具函数，例如大小端转换、数值压缩。

这一层虽然不起眼，但很多模块都依赖它。

## 5. 关键文件逐个怎么理解

### 5.1 `main.c`

最简入口文件。

重点理解：

- 为什么要先 `System_Init()` 再循环 `System_RunOnce()`
- 为什么初始化失败后进入死循环

这类结构在裸机嵌入式里非常常见。

### 5.2 `app/system.h`

重点看 `SystemContext`。

这是一个“总状态包”。

很多函数不喜欢用一大堆全局变量，所以把系统运行所需内容都装到这个结构体里，再把指针传来传去。

### 5.3 `app/system.c`

这是最值得反复看的文件。

你可以重点盯住几个函数：

- `System_Init()`
- `System_RunOnce()`
- `System_OnCanRx()`
- `System_OnShortTripInterrupt()`

如果你能把这 4 个函数的调用链看懂，这个项目主干你就懂一半了。

### 5.4 `protection/protection.c`

这个文件体现了“状态机”的思想。

系统状态可能有：

- `SSCB_STATE_NORMAL`
- `SSCB_STATE_FAULT_ACTIVE`
- `SSCB_STATE_RECOVER_WAIT`
- `SSCB_STATE_RECOVER_TRY`
- `SSCB_STATE_LOCKOUT`

重点不是死记名字，而是理解状态如何切换：

1. 正常运行
2. 发现故障
3. 跳闸
4. 等待恢复条件
5. 尝试恢复或进入锁定

### 5.5 `protocol/protocol_service.c`

这个文件很像“报文调度中心”。

收到帧后，代码会依次判断：

1. 是不是时间同步命令
2. 是不是控制命令
3. 是不是参数读写命令

然后决定调用哪个业务动作。

这是初学者理解“协议分发”的好例子。

### 5.6 `storage/param_store.c`

这是理解“参数管理”的关键文件。

可以重点看：

- `ParamStore_Defaults()`
- `ParamStore_Load()`
- `ParamStore_Save()`
- `ParamStore_Get()`
- `ParamStore_Set()`

理解它之后，你会知道：

- 参数默认值从哪里来
- 为什么要做 CRC
- 为什么要检查版本号和 magic
- 上位机写参数后为什么能掉电保存

## 6. 上位机代码怎么理解

`host/` 目录是 C# 代码，不是 MCU 固件。

主要分三块：

- `host/SscbHost.Core`
  核心协议和 CAN 抽象层
- `host/SscbHost.WinForms`
  Windows 界面程序
- `host/SscbHost.Tests`
  协议测试

建议阅读顺序：

1. `host/SscbHost.Core/Protocol/SscbConstants.cs`
2. `host/SscbHost.Core/Protocol/SscbModels.cs`
3. `host/SscbHost.Core/Protocol/SscbProtocol.cs`
4. `host/SscbHost.WinForms/MainForm.cs`
5. `host/SscbHost.Tests/SscbProtocolTests.cs`

这样你能先知道协议定义，再知道界面如何使用协议，最后看测试如何验证协议。

## 7. 代码里一些常见写法是什么意思

### 7.1 `ctx == 0`

这就是在判断指针是不是空指针。

很多 C 代码喜欢写 `0`，等价于 `NULL`。

### 7.2 `(void)函数名(...)`

表示“我故意不使用返回值”。

例如：

```c
(void)FaultLog_Clear();
```

意思是：

- 我调用了这个函数。
- 但我现在不处理它的返回结果。

### 7.3 `for (;;)`

这是 C 语言里最典型的死循环写法，等价于：

```c
while (1)
```

### 7.4 `static`

在这个项目里常见两种用法：

1. `static` 局部变量
   例如 `main.c` 里的 `static SystemContext system;`
   含义是它只初始化一次，并且生命周期贯穿整个程序。

2. `static` 函数
   例如 `app/system.c` 里的 `static void tx_frame(...)`
   含义是这个函数只给当前 `.c` 文件内部使用，不暴露给外部。

## 8. 你下一步应该怎么学

建议你边看边做这 4 件事：

1. 先只看 `main.c`、`app/system.c`，画出函数调用关系。
2. 再看 `protection/protection.c`，画状态切换图。
3. 再看 `protocol/protocol_service.c`，整理 CAN 命令分发流程。
4. 最后看 `param_store.c`，理解参数保存与校验。

如果你一开始就试图逐字符读完全部代码，效率会很低。先抓主流程，再看细节，会更容易理解。

## 9. 关于“每行注释”

真正工程里，通常不会给每一行都写注释，因为：

1. 注释过多会把代码本身淹没。
2. 一旦代码修改，注释很容易过期。
3. 好的方式是：
   - 核心流程写清楚
   - 难懂逻辑写清楚
   - 配合一份整体讲解文档

所以我这次采用的是更实用的方式：

1. 给核心入口代码补中文注释。
2. 额外提供这份源码入门文档，帮助你按模块理解项目。

如果你愿意继续，我下一轮可以继续做两件更细的事：

1. 继续给 `protection/`、`protocol/`、`storage/` 里的每个函数补“块级中文注释”。
2. 单独给你做一份“逐文件逐函数白话解释版”。
