## 2. 项目总体目标

基于 TI **TMS320F280049C / F280049CPQZ** 平台，完成智能固态断路器控制器软件开发，实现以下核心能力：

1. 电压、电流、温度实时采样与工程量换算
2. CAN 周期上报与上位机命令交互
3. CMPSS 短路硬件快速保护
4. 基于 RMS + I²t 的过载保护
5. 故障记录、时间戳管理与主动上报
6. 参数可配置、掉电保存、故障记录掉电保存
7. 基础自检与低功耗支持
8. 满足联调测试与最终验收要求

## 4. 软件总体架构规划

### 4.1 分层结构

建议沿用需求文档给出的工程化分层：

```text
/app
/bsp
/driver
/protection
/protocol
/storage
/selftest
/common
```

### 4.2 分层职责规划

#### `/app`

负责系统主流程组织，是各功能模块的调度入口：

- 系统初始化流程
- 周期任务调度
- 主状态机管理
- 保护动作后的系统联动
- 模块间数据流组织

#### `/bsp`

负责芯片与板级初始化：

- 时钟初始化
- GPIO 初始化
- 中断向量配置
- 外设基础启动顺序
- Idle 模式入口管理

#### `/driver`

负责外设驱动抽象：

- ADC 驱动
- ePWM 驱动
- CMPSS 驱动
- CAN 驱动
- SPI / FRAM 驱动
- 定时器 / 时间基准驱动

#### `/protection`

负责保护算法与故障动作：

- 短路保护软件联动
- 过载保护 I²t 算法
- 过压/过温判定
- 采样异常判定
- 故障恢复与闭锁逻辑

#### `/protocol`

负责通信协议：

- CAN 报文 ID 规划
- 实时数据打包
- 故障事件上报
- 时间同步处理
- 参数读写命令处理
- 控制命令处理

#### `/storage`

负责掉电保存：

- FRAM 驱动封装
- 参数区管理
- 故障记录区管理
- CRC / 完整性校验
- 上电加载与默认参数恢复

#### `/selftest`

负责上电与运行期自检：

- 采样链路异常粗判
- NTC 开短路检查
- ADC 范围检查
- CMPSS 配置一致性检查
- 自检结果上报

#### `/common`

负责公共支撑：

- 数据类型定义
- 参数结构体定义
- 故障码 / 状态码定义
- 时间戳结构
- 环形缓冲区
- 工具函数与宏定义

## 10. 风险与控制点

## 10.1 技术风险

### 短路保护链路风险

风险点：CMPSS、ePWM Trip、驱动链路联动不符合预期。  
控制措施：优先完成硬件快速关断闭环测试，不等待全部功能开发完成。

### 采样精度与周期风险

风险点：20 us 采样与 10 ms 算法调度冲突。  
控制措施：尽早建立节拍与中断优先级设计，先保保护，再保通信。

### FRAM 数据一致性风险

风险点：参数写入异常、掉电中断写入导致数据损坏。  
控制措施：增加 CRC、双阶段写入或写前校验策略。

### 协议联调风险

风险点：上位机与固件对 CAN 报文理解不一致。  
控制措施：在阶段4前先冻结协议定义与参数编码方式。

## 10.2 项目管理风险

### 需求边界蔓延

风险点：开发过程中临时加入未定义功能。  
控制措施：严格以当前需求版本为基线管理，新增项单独评审。

### 联调滞后

风险点：全部代码写完后才集中联调。  
控制措施：按阶段逐步联调，采用“驱动—采样—保护—协议—存储”顺序推进。

## 12. 建议的实施顺序

建议按以下优先级推进：

```text
1. 工程框架与驱动初始化
2. ADC采样 + 时间基准
3. CMPSS + ePWM Trip短路链路
4. RMS + I²t过载算法
5. CAN上报与命令协议
6. FRAM参数与故障记录
7. 状态机/恢复/闭锁
8. 自检与低功耗优化
9. 联调、回归、验收
```

该顺序的核心原则是：

- 优先实现保护安全链路
- 先打通主链路，再做边界功能
- 先保证正确性，再考虑优化
